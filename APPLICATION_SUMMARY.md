## High-level purpose

The application is a browser-based resident case-log analysis tool. It is designed to ingest a combined resident case-log PDF, split it into resident-level records, compare each resident against minimum case requirements, and generate both on-screen analysis and downloadable PDF exports.

The app is currently oriented around a BWH / MGB neurosurgery workflow and specifically supports the combined ACGME-style resident log PDF format now in use.

## Main user workflow

The main user flow starts in [src/pages/Index.tsx](/Users/andrew/Desktop/ResidentLogging/ResidentTracker/resident-success-tracker-copy-2026-03-16/src/pages/Index.tsx).

1. The user uploads one or more files through the uploader component.
2. Each file is parsed through `parseResidentFile`.
3. A single combined PDF can yield multiple resident records.
4. Each parsed resident is assigned a PGY year through the resident roster lookup.
5. Each resident record is converted into comparison data.
6. The comparison data is ranked and displayed on screen.
7. The user can export:
   - a grid-style BWH PDF
   - a per-resident cohort comparison PDF

## Input handling

The parsing entry point is [src/lib/parseResidentData.ts](/Users/andrew/Desktop/ResidentLogging/ResidentTracker/resident-success-tracker-copy-2026-03-16/src/lib/parseResidentData.ts).

### Supported file types

The parser still contains support for both:

- Excel files
- PDF files

However, the current intended workflow is the combined PDF case-log file.

### Combined PDF parsing

The combined PDF logic works as follows:

1. The PDF is read with `pdfjs-dist`.
2. All text items are extracted with X/Y coordinates.
3. Those items are grouped into rows by vertical position.
4. The parser finds resident section boundaries by looking for rows that begin with `Resident:`.
5. Each resident section is parsed independently into a `ResidentData` object.

This allows a single PDF to contain many residents, including residents whose report spans multiple pages.

### Resident-level fields extracted

Each parsed resident record includes:

- `residentName`
- `program`
- `asOfDate`
- `categories`

Each category includes:

- category name
- lead resident surgeon count
- senior resident surgeon count
- lead + senior total
- lead minimum
- lead + senior minimum

## Comparison logic

Comparison and ranking logic lives in [src/lib/compareResidents.ts](/Users/andrew/Desktop/ResidentLogging/ResidentTracker/resident-success-tracker-copy-2026-03-16/src/lib/compareResidents.ts).

For each resident:

- lead-only comparisons are calculated
- lead-plus-senior comparisons are calculated

Each category is classified into one of:

- `met`
- `below`
- `critical`
- `no_minimum`

Rows with zero minimums are preserved for output fidelity, but they are excluded from overall completion scoring.

The overall ranking is based on the average of:

- overall lead percentage
- overall lead-plus-senior percentage

## Export types

There are two primary PDF exports currently exposed in the UI.

### 1. Export Grid

This export is generated by [src/lib/exportBwhTemplatePdf.ts](/Users/andrew/Desktop/ResidentLogging/ResidentTracker/resident-success-tracker-copy-2026-03-16/src/lib/exportBwhTemplatePdf.ts).

Purpose:

- produce a single-page grid aligned with the BWH-style template layout
- display category rows, program mean, minimums, PGY-specific minimums, and resident columns

How it works:

- it uses a hardcoded row map corresponding to the BWH template structure
- it uses a category normalization layer so parsed PDF category names map reliably to template rows
- residents are grouped by PGY
- values are pulled from lead-plus-senior totals

Visual logic:

- red for below minimum
- yellow for below mean but above minimum
- green for at or above mean
- gray for not uploaded

The grid exporter also contains:

- a fallback default PGY grouping
- initials mappings for narrow resident columns
- PGY-specific minimum tables used in the output

### 2. Export Per-Resident Performance PDF

This export is generated by [src/lib/exportPdf.ts](/Users/andrew/Desktop/ResidentLogging/ResidentTracker/resident-success-tracker-copy-2026-03-16/src/lib/exportPdf.ts).

Purpose:

- generate a two-page report per resident
- show category-by-category comparisons using bars and markers

Current behavior:

- page 1 is lead + senior
- page 2 is lead only

Important current cohort behavior:

- the cohort mean and cohort range are now calculated using the resident's own PGY class, not the full uploaded resident set
- if a resident has no PGY assigned, the full uploaded set is used as a fallback

Visual encoding:

- gray bar segment = PGY cohort range
- red vertical line = PGY cohort mean
- black vertical line = minimum requirement
- blue triangle = current resident

Recent precision behavior:

- the displayed mean and range text now use the same numeric precision as the plotted values
- the mean marker is drawn last so it remains visible even if it overlaps the resident marker

If the application needs to be updated in the future, the most likely maintenance tasks are:

2. Adjust category normalization if the source PDF wording changes.
3. Update template row mappings if the BWH grid format changes.
4. Recheck export labeling if cohort comparison rules change again.
